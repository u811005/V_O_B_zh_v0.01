# AGENTS.md

このリポジトリで作業するエージェント向けの作業ルールです。プロジェクトの実態に合わせて、小さく安全に変更してください。

## 基本方針

- 返答・ドキュメントは、ユーザーから別指定がない限り日本語で書く。
- 既存の HTML / CSS / JavaScript 構成を尊重する。新しいビルドツール、フレームワーク、依存関係は、必要性が明確な場合だけ導入する。
- 作業前に既存ファイルを読み、同じ責務のファイルへ変更を入れる。似た機能を別の場所に増やさない。
- 未コミット変更や削除済みファイルは、ユーザーまたは別作業のものとして扱う。依頼されていない変更を戻さない。
- 日本語テキストを扱うため、UTF-8 と LF 改行を維持する。PowerShell 表示だけの文字化けを見て再エンコードしない。

## 最初に読む文書

- `PROJECT_OVERVIEW.md`: ゲーム内容と主要システムの全体像。
- `WORLDVIEW.md`: 世界観、ストーリー、外部勢力、今後の新要素追加時の設定指針。
- `ARCHITECTURE.md`: ファイル分割、状態管理、ターン進行の構造。
- `DIALOGUE_RULES.md`: 会話・セリフ追加時の判断基準。
- `CODE_REVIEW.md`: レビュー時の観点と検証手順。
- `GAME_GUIDE_JA.md` / `README.md` / `Readme.txt`: プレイヤー向け説明と背景設定。
- `docs/dialogue-refactor-handoff.md`: 会話エンジンの詳細な引き継ぎ。

## 実装ルール

- アプリの入口は `index.html` と `js/app.js`。`js/app.js` は各モジュールをつなぎ、ブラウザ上のボタン操作を `window` に公開する。
- 共有状態は `js/main.js` の `theVillage` と、`js/classes.js` の `Village` / `Villager` が中心。保存互換性に関わるフィールド名は安易に変えない。
- 画面更新は原則 `updateUI(theVillage)` を通す。DOM を直接更新する場合は、そのモーダルや表示部品の責務に閉じ込める。
- 仕事・行動の候補更新は `js/domain/jobTables.js` を通す。襲撃中の特殊行動は `js/raidRules.js` の判定と揃える。
- 仕事・行動の体力/メンタル消費計算と副作用のない労働成果計算は `js/domain/jobMath.js` を通す。`jobs.js` と `ui.js` に同じ計算式を重複実装しない。
- データ追加は `js/data/` 配下へ寄せる。ロジックファイルへ大量の固定文言や表データを直書きしない。
- 世界観やストーリーに関わる新要素を追加する場合は、`WORLDVIEW.md` の方向性に合うか確認する。設定から外れる場合は、理由を明確にし、必要に応じて同文書も更新する。
- 新しい仕事、行動、資源、建築、奇跡、状態異常、特性、イベント、襲撃者、ルールなどを追加したり、既存仕様を変更した場合は、必要に応じて `js/data/dictionaryData.js` の用語説明も追加・更新する。
- 新機能、仕様変更、ファイル構成変更、会話ルール変更、レビュー手順変更などを行った場合は、影響する Markdown ドキュメントも必要に応じて更新する。小さなバグ修正や内部実装だけの変更で、仕様・使い方・構造説明に影響しない場合は更新不要。
- ランダム性が絡む処理では、境界値、空配列、対象者がいない場合を必ず考える。

## 会話・セリフの扱い

- 通常会話は `js/dialogue/dialogueEngine.js` と `js/data/dialogue/` 配下を優先する。
- 口調は肉体ではなく、精神側の性別・年齢・特性を優先して解決する。
- 新規セリフは、説明文ではなく「本人の発話」として書く。
- 現代的すぎる語彙やメタ表現は避け、村、祭、畑、祈り、魔素、神、奇跡など既存の世界観に寄せる。
- 詳細なルールは `DIALOGUE_RULES.md` と `docs/dialogue-refactor-handoff.md` を参照する。

## 検証

- JavaScript を変更したら、少なくとも変更ファイルに対して `node --check` 相当の構文確認を行う。
- ターン進行、保存/読込、会話、建築、奇跡、襲撃に触れた場合は、ブラウザで該当導線を確認する。
- UI を変更したら、PC 表示と `スマホ仮` 表示の両方で、文字のはみ出し、ボタンの押しやすさ、表の崩れを確認する。
- 新要素や仕様変更を行った場合は、用語検索で説明すべき内容が `js/data/dictionaryData.js` に反映されているか確認する。
- 保存形式に触れた場合は、古い JSON / localStorage データを読み込めるかを確認する。

## 変更の粒度

- 依頼範囲に必要な変更だけを行う。
- ついでの大規模整理、命名変更、責務移動は避ける。
- どうしても広い整理が必要な場合は、理由、影響範囲、検証方法を明記してから進める。
